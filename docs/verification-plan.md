# Service Sheet Generation — Verification Plan

Ordered **fail-fast**: cheapest automated checks first, expensive manual checks last.

---

## Phase 1: Static Analysis (< 30s, blocks everything)

1. **Type check**: `npm run typecheck` — zero errors
2. **Lint check**: `npm run lint` — zero violations
3. **Build check**: `npm run build` — succeeds (catches server component issues)

## Phase 2: Unit Tests (< 60s, automated)

4. **PDF renders without crash**: Call `ServiceSheetDocument({ data })` with a complete `ServiceSheetData` fixture for each of the 7 service types — assert valid React element (no throw)
5. **DOCX returns valid buffer**: Call `generateServiceSheetDocx(data)` — assert `Buffer` with length > 0, starts with DOCX magic bytes (`PK` zip header, `0x504B0304`)
6. **Service type label coverage**: Assert every `ServiceType` enum value has an entry in `serviceTypeLabel` in both `service-sheet.tsx` and `service-sheet-docx.ts` — catches silent fallback to raw enum strings
7. **Slot label coverage**: Assert every `MusicSlotType` enum value has an entry in `MUSIC_SLOT_LABELS` — catches raw enum display for known slot types
8. **Data shape contract**: Mock DB responses, run the `sheetData` construction logic (sheet/route.ts lines 56–73), assert resulting `ServiceSheetData` matches expected shape

## Phase 3: Integration / API Tests (< 2min, automated)

9. **PDF endpoint**: `GET /api/churches/{id}/services/{id}/sheet` → 200, `Content-Type: application/pdf`, `Content-Disposition: attachment`
10. **DOCX endpoint**: `?format=docx` → 200, `Content-Type: application/vnd.openxmlformats-officedocument.wordprocessingml.document`
11. **404 for missing service**: Invalid `serviceId` → 404 (not 500)
12. **Auth enforcement**: No session → 401; non-member → 403
13. **Invalid format param**: `?format=txt` → verify falls through to PDF default or returns 400

## Phase 4: Edge Case / Robustness Tests (automated)

14. **Empty arrays**: `readings: []`, `musicSlots: []` → renders header + footer only, no empty section headers visible
15. **Missing optional fields**: `collect: undefined`, `logoUrl: undefined`, `notes: undefined` on all slots → no literal `"undefined"` text in output
16. **Long text overflow**: 2000-char collect, 50+ music slots, 10 readings → does not crash, paginates correctly, footer appears on every page (not just page 1)
17. **Special characters**: `&`, `<`, `>`, `"`, `'`, em-dash `—`, curly quotes `""''`, Unicode (`ñ`, `ö`, `ä`) in church name, slot values, reading references → no rendering errors or mojibake
18. **All 7 service types**: One sheet per `ServiceType` enum value → all render, all show correct human-readable label (not raw enum)
19. **A5 format**: `format: "A5"` → smaller page style applied (30px padding, 10pt font vs A4's 40px/11pt)

## Phase 5: Visual / Manual Verification (human review)

> Only reached after all automated checks pass.

20. **Sung Eucharist PDF** — open and verify:
    - Header: church name (bold 14pt), service title (bold 18pt), liturgical day, date, season/colour badge
    - Collect: italic, 1.4 line height
    - Readings: position labels italic, references regular weight
    - Music: slot labels bold, values regular, notes in parentheses
    - Footer: "Generated by Precentor" centered with top border
    - Liturgical colour accent on cover *(when implemented)*
    - Page numbers *(when implemented)*
    - CCLI notice from `churches.ccliNumber` *(when implemented)*

21. **Choral Evensong PDF** — verify slot ordering:
    - Responses → Psalm → Magnificat → Nunc Dimittis → Anthem → Office Hymn → Final Hymn → Voluntaries
    - BCP structure labels *(when implemented)*

22. **DOCX parity** — open PDF and DOCX side by side for same service:
    - Same sections in same order
    - Times New Roman throughout
    - Bold/italic/size parity (e.g. church name 28 half-points = 14pt)
    - Notes in italic smaller text (18 half-points in DOCX)
    - Border separators present (header bottom, footer top)

23. **Summary sheets** *(when implemented)*:
    - Section groupings by date/service
    - Compact tabular layout
    - Print-friendly (no cut-off columns)

## Phase 6: Print & Compatibility

24. **Print preview**: PDF in browser print dialog → no clipping, correct A4 margins
25. **DOCX in Word/LibreOffice**: Opens without "repair" prompt, formatting intact
26. **PDF in multiple readers**: Chrome, Preview (macOS), Adobe Reader → no font substitution warnings
